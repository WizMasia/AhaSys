# DOCS GUIDE

## OVERVIEW
Docs hold setup notes, design/refactor records, and the benchmark corpus/report surface. Some files are generated by benchmark runs.

## STRUCTURE
```
docs/
├── benchmark/                         # benchmark README, summary JSON, generated case files
├── images/demo.png                    # README screenshot
├── ocr-setup.md                       # EasyOCR setup and troubleshooting
├── refactor-modular-audit-flow-summary.md
├── refactoring_specification.md
└── ponytail_simplification.md
```

## WHERE TO LOOK
| Task | Location | Notes |
|------|----------|-------|
| OCR operator setup | `ocr-setup.md` | Python 3.10-3.12, EasyOCR/PyTorch, env vars. |
| Refactor constraints | `refactoring_specification.md` | Bans `any`, `ts-ignore`, hardcoded values, duplicate JSON-repair logic. |
| Known follow-ups | `refactor-modular-audit-flow-summary.md` | Includes OCR fallback semantic notes. |
| Benchmark docs/data | `benchmark/README.md`, `benchmark/test_runs.json`, `benchmark/cases/` | May be generated by server benchmark runner. |
| README image | `images/demo.png` | Keep path stable for root README. |

## CONVENTIONS
- Keep operational docs aligned with `.env.example`, `Dockerfile`, and `docker-compose.yml`.
- When changing OCR behavior, update `ocr-setup.md` and root README together.
- OCR docs mention `OCR_ENABLED`, `OCR_PYTHON_BIN`, `EASYOCR_LANGS`, `EASYOCR_GPU`, and `EASYOCR_MODEL_STORAGE`.
- Treat `docs/benchmark/cases/` as benchmark output/corpus, not hand-authored app logic.
- Preserve Korean product/domain wording where it is user-facing.

## ANTI-PATTERNS
- Do not document Render as having OCR enabled; current deployment guidance disables it.
- Do not let refactor docs drift from actual commands: `npm run lint`, `npm test`, `npm run build`.
- Do not overwrite benchmark outputs unless the benchmark run is the task.
- Do not add generic project-management prose; keep docs tied to code behavior.
